# 论点验证功能需求文档

**项目名称:** AI News Radar - 论点验证模块
**版本:** v1.0
**日期:** 2026-03-01
**状态:** 草案待确认

---

## 1. 项目概述

### 1.1 背景

当前 AI News Radar 系统已具备：
- 10个高质量 AI/科技新闻源的实时聚合
- 24小时新闻更新监控
- 历史数据归档（archive.json）
- 中英双语标题支持

**新增需求：**
用户希望能够提出论点或声明，系统基于这些新闻源进行深度分析，给出证明或证伪的结论，并提供证据链支持。

### 1.2 目标

**主要目标：**
- 为用户提供基于事实的论点验证能力
- 利用现有新闻数据，减少信息验证成本
- 提供可信的证据溯源

**次要目标：**
- 提升用户对 AI/科技趋势的理解
- 辅助决策制定
- 发现虚假信息和谣言

### 1.3 范围

**包含：**
- 论点输入和理解
- 基于时间窗口的证据检索
- AI 驱动的分析和推理
- 可信度评分
- 证据链展示

**不包含：**
- 实时事实核查（基于已有数据，非实时抓取）
- 非 AI/科技领域的验证
- 知识图谱（Phase 2）

## 2. 用户故事

### 2.1 核心用户故事

**US-01: 验证科技传闻**
> 作为 AI 从业者，我想要验证"某公司将在某时间发布某产品"的传闻，以便判断是否值得关注和准备。

**US-02: 分析市场趋势**
> 作为投资者，我想要验证"某技术将在明年成为主流"的论点，以便做出投资决策。

**US-03: 核实行业动态**
> 作为分析师，我想要验证"某公司收购了另一家公司"的消息，以便撰写准确报告。

**US-04: 对比观点冲突**
> 作为研究者，我想要对比"专家 A 说 X，专家 B 说 Y"的冲突观点，以便了解真相。

### 2.2 使用场景示例

**场景 1: 产品发布验证**
```
用户输入: "GPT-5 将在 2025 年发布"

系统分析:
- 检索时间: 2025-03-01 前 7 天的新闻
- 支持证据: OpenAI 官方博客暗示进展（2条）
- 反对证据: Sam Altman 推特否认时间表（1条）
- 中性信息: 行业分析师预测（8条）

结论: "无法确定 - 信息矛盾"
置信度: 35% (低)
```

**场景 2: 技术趋势判断**
```
用户输入: "多模态 AI 将在 2025 年成为主流"

系统分析:
- 检索时间: 2025-03-01 前 30 天的新闻
- 支持证据: Google Gemini、GPT-4V 发布（15条）
- 市场数据: 多模态应用增长 300%（5条）
- 专家观点: 8/10 专家表示认同（3条）

结论: "高度可能 - 多方支持"
置信度: 85% (高)
```

## 3. 功能需求

### 3.1 MVP 功能 (Phase 1 - Week 1-2)

#### F-01: 论点输入界面
**描述:** 用户友好的论点输入界面

**需求:**
- [ ] 文本输入框（支持多行）
- [ ] 时间范围选择器（1天、3天、7天、30天、自定义）
- [ ] 可信度阈值滑块（只显示高于某置信度的结果）
- [ ] 提交按钮
- [ ] 历史记录入口

**优先级:** P0

---

#### F-02: 论点理解模块
**描述:** 解析用户输入，提取关键信息

**需求:**
- [ ] 关键词提取（GPT-5, 发布, 2025）
- [ ] 实体识别（公司、产品、人物、技术）
- [ ] 事件类型识别（发布、收购、合作、争议）
- [ ] 时间解析（2025年 → 2025-01-01 至 2025-12-31）

**技术方案:**
- 使用 NLP 库（spaCy 或 LLM）
- 输出结构化的 Query 对象

**优先级:** P0

---

#### F-03: 证据检索模块
**描述:** 从新闻数据中检索相关证据

**需求:**
- [ ] 关键词匹配检索
- [ ] 时间窗口过滤
- [ ] 相关性排序
- [ ] 去重处理

**检索策略:**
1. **关键词检索:** 在标题、摘要中匹配关键词
2. **时间过滤:** 只返回指定时间范围内的新闻
3. **相关性评分:** TF-IDF + 关键词密度

**数据源:**
- `data/latest-24h.json` (最近24h)
- `data/archive.json` (历史归档)

**优先级:** P0

---

#### F-04: AI 分析引擎
**描述:** 使用 LLM 对证据进行推理分析

**需求:**
- [ ] 证据聚合和分类
- [ ] 支持证据提取
- [ ] 反对证据提取
- [ ] 矛盾检测
- [ ] 结论生成

**LLM Prompt 模板:**
```
你是一个事实核查专家。请基于以下证据分析给定的论点。

论点: {claim}

证据列表:
{evidence}

请分析:
1. 有多少证据支持这个论点？
2. 有多少证据反对这个论点？
3. 证据之间是否存在矛盾？
4. 给出你的结论（证明/证伪/无法判断）
5. 评估你的置信度（0-100%）

输出格式:
{
  "verdict": "SUPPORTED" | "REFUTED" | "INCONCLUSIVE",
  "confidence": 0-100,
  "supporting_evidence": [evidence_ids],
  "refuting_evidence": [evidence_ids],
  "reasoning": "详细推理过程"
}
```

**模型选择:**
- MVP: GPT-4o mini ($0.15/1M tokens) - 成本优化
- 增强版: Claude 3.5 Sonnet - 更高质量

**优先级:** P0

---

**UI 组件:**

**优先级:** P0

#### F-07: 可信度评分算法
**描述:** 多维度评估证据可信度

**评分因素:**
- [ ] 信源权威性（官方博客 > 媒体 > 社交）
- [ ] 证据数量（更多证据 = 更高分）
- [ ] 证据一致性（无矛盾 = 更高分）
- [ ] 时间新鲜度（近期 = 更高分）
- [ ] 独立性（多个独立信源 = 更高分）

**计算公式:**
```
confidence = (
  source_authority * 0.3 +
  evidence_count * 0.25 +
  consistency_score * 0.2 +
  freshness_score * 0.15 +
  independence_score * 0.1
) * 100
```

**优先级:** P1

## 6. 数据模型

### 6.1 论点 (Claim)

```python
@dataclass
class Claim:
    id: str                          # UUID
    text: str                        # 原始文本
    keywords: List[str]              # 提取的关键词
    entities: List[str]              # 识别的实体
    event_type: str                  # 事件类型
    time_window: int                 # 时间窗口（天数）
    created_at: datetime             # 创建时间
```

### 6.2 证据 (Evidence)

```python
@dataclass
class Evidence:
    id: str                          # 新闻 ID
    title: str                       # 标题
    source: str                      # 来源网站
    url: str                         # 链接
    published_at: datetime           # 发布时间
    summary: str                     # 摘要
    relevance_score: float           # 相关性分数 0-1
    sentiment: str                   # positive/negative/neutral
```

### 6.3 分析结果 (AnalysisResult)

```python
@dataclass
class AnalysisResult:
    claim_id: str
    verdict: Verdict                 # SUPPORTED | REFUTED | INCONCLUSIVE
    confidence: float                # 0-100
    supporting_evidence: List[Evidence]
    refuting_evidence: List[Evidence]
    neutral_evidence: List[Evidence]
    reasoning: str                   # LLM 推理过程
    evidence_count: int              # 总证据数
    created_at: datetime
```

### 6.4 数据库 Schema (SQLite)

```sql
-- 论点表
CREATE TABLE claims (
    id TEXT PRIMARY KEY,
    text TEXT NOT NULL,
    keywords TEXT,                   -- JSON array
    entities TEXT,                   -- JSON array
    time_window INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 分析结果表
CREATE TABLE analysis_results (
    id TEXT PRIMARY KEY,
    claim_id TEXT REFERENCES claims(id),
    verdict TEXT NOT NULL,
    confidence REAL NOT NULL,
    reasoning TEXT,
    evidence_count INTEGER,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 证据关联表
CREATE TABLE claim_evidence (
    claim_id TEXT REFERENCES claims(id),
    evidence_id TEXT,
    evidence_type TEXT,              -- supporting | refuting | neutral
    relevance_score REAL,
    PRIMARY KEY (claim_id, evidence_id)
);
```

### 7.2 错误处理

**错误响应格式:**
```json
{
  "error": {
    "code": "INVALID_CLAIM",
    "message": "论点文本不能为空",
    "details": {}
  }
}
```

**错误码:**
- `INVALID_CLAIM`: 论点无效
- `TIME_WINDOW_EXCEEDED`: 时间窗口超出范围
- `NO_EVIDENCE_FOUND`: 未找到相关证据
- `LLM_API_ERROR`: LLM API 调用失败
- `INTERNAL_ERROR`: 内部错误

## 8. 验收标准

### 8.1 功能验收

#### AC-01: 论点验证
**Given:** 用户输入一个有效的论点
**When:** 点击"开始验证"
**Then:**
- 系统在 10 秒内返回结果
- 结果包含结论、置信度、证据链
- 至少返回 1 条相关证据（如果有）

#### AC-02: 证据相关性
**Given:** 系统返回证据列表
**When:** 用户查看证据
**Then:**
- 至少 80% 的证据与论点相关
- 证据按相关性排序
- 显示证据来源和时间

#### AC-03: 置信度准确性
**Given:** 同一论点被多次验证
**When:** 证据库更新后重新验证
**Then:**
- 置信度分数会根据新证据变化
- 结论可能改变

#### AC-04: 历史记录
**Given:** 用户已完成多次验证
**When:** 查看历史记录
**Then:**
- 显示所有历史验证
- 可以重新查看结果
- 可以删除记录
